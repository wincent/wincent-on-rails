  1. Greg Hurrell 2010-10-15T15:52:53Z

    As I replied to the original comment:

    This is caused by someone or something hitting the feed under the "rails.wincent.com" host (which is just an alias), and the feed then gets cached with all the URLs pointing at rails.wincent.com instead of wincent.com, ruining the feed for everybody.

    I actually fixed this a long time ago by explicitly forcing it to always use wincent.com as the host regardless of what host is used in the request, but it looks like something in Rails 3 has broken that. Oddly, the spec suite, which explicitly tests this, passes fine.

    I'll investigate and try to re-fix the issue again, but in any case I think it's about time the old rails.wincent.com was decommissioned anyway.

    Ok, so this is now effectively mitigated because I've set up a rewrite rule to permanently redirect any request to https://rails.wincent.com/foo to redirect to https://wincent.com/foo (this was already in place for HTTP requests to rails.wincent.com). This means that it should be impossible to touch the Rails app at all without going through "wincent.com" as the host, so cached feeds should no longer get polluted in this way.

    I've blown away all cached feeds, so newly-requested feeds should be fine from here on. (Note that if you try to look at the feed in Safari you might still see the old version of the feed, as Safari, or perhaps PubSubAgent, does some caching of its own.)

    Will leave this ticket open however as I'd still like to investigate the root cause.
